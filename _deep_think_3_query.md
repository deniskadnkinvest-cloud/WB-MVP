## 📌 Контекст ситуации

### О проекте
**Seller Studio** — это SaaS веб-приложение для селлеров (продавцов) маркетплейсов Wildberries, Ozon и др. Позволяет генерировать профессиональные фотографии товаров (одежды) на AI-моделях за 30 секунд вместо реальных фотосессий за ₽15,000–50,000.

**Технология:** Virtual Try-On (VTON) — селлер загружает фото одежды → выбирает модель, позу, фон → AI генерирует студийное фото, где модель одета в эту одежду.

### Техстек и архитектура

**Фронтенд:**
- React 19 + Vite 8 + Framer Motion
- Деплой: Vercel
- URL продакшена: `https://vton-mvp-omega.vercel.app`
- Telegram Mini App (бот запускает WebApp через кнопку)

**Бэкенд:**
- Vercel Serverless Functions (Node.js)
- Основные API-эндпоинты:
  - `POST /api/generate-image` — главная генерация (maxDuration: 120s)
  - `POST /api/create-payment` — создание инвойса Telegram Stars (maxDuration: 10s)
  - `POST /api/payment-webhook` — вебхук от Telegram для подтверждения оплаты (maxDuration: 30s)
  - `POST /api/verify-panx-token` — верификация токена партнёрского API (maxDuration: 10s)

**AI-движок генерации:**
- Основной: **KIE.ai** (модель `nano-banana-2`) — обёртка над Gemini для VTON
- API-ключ: `KIE_API_KEY` (единственный источник, без фолбеков)
- Поток: Клиент → Vercel Serverless → KIE.ai API (createTask → polling → resultUrl → download → base64 → клиенту)
- Вспомогательный: **Google Gemini API** (`gemini-2.5-flash`) — используется ТОЛЬКО для:
  - AI-классификации одежды (определение категории, цвета, фита) в Inngest pipeline
  - AI Quality Score (LLM-as-a-Judge: проверка анатомии, подсчёт пальцев, фиделити одежды)

**БД и хранилище (Firebase):**
- **Firebase Auth** — Google Sign-In авторизация
- **Firestore** — структура данных:
  - `users/{uid}/subscription/current` — подписка: plan, credits, creditsTotal, planActivatedAt, planExpiresAt, payments[]
  - `users/{uid}/saved_models/{modelId}` — сохранённые модели: name, type, imageUrls[], prompt, createdAt
  - `users/{uid}/saved_locations/{locationId}` — сохранённые локации: title, imageUrls[], thumbnail, prompt, createdAt
- **Firebase Storage** — хранение изображений моделей и локаций: `users/{uid}/models/`, `users/{uid}/locations/`

**Async pipeline (Inngest):**
- Fan-Out паттерн для пакетной обработки (Auto-Catalog)
- Steps: classify → map-parameters → generate → quality-score → save-results
- Concurrency: 5 параллельных генераций
- Retries: 2 авто-ретрая при сбоях

**Оплата:**
- Telegram Stars через Bot API
- Инвойс создаётся через `createInvoiceLink`, подтверждение через webhook `successful_payment`

### Функциональность приложения

**Fashion Mode (основной):**
- Загрузка одежды (drag-and-drop, до 9 вещей)
- Кастинг-рум: пресеты моделей по полу, этничности, телосложению
- Детализация: волосы, макияж, тату, пирсинг, телосложение
- Кастомные текстовые промпты для модели
- "Мои модели" (Identity Preservation — сохранение лица)
- Позы (50 женских + 50 мужских пресетов + свободный текст)
- Ракурсы камеры (full body, medium shot, close-up)
- Фоны/Локации (пресеты + "Мои локации" + свободный текст)
- Форматы (1:1, 3:4, 4:3, 9:16, 16:9)
- Фотосессия (пакетная генерация 3 или 5 фото)
- Итеративное редактирование (Photo Edit Mode)
- Beauty Mode / Ultra-Realism Mode
- Adaptive Fashion (инклюзивная мода)

**Product Mode (предметная съёмка):**
- Категории: косметика, парфюм, ювелирка, БАДы, декор/свечи, электроника, зоотовары, спорт, чехлы, автотовары + "Другое"
- Композиции, фоны, эффекты для каждой категории
- Опционально: с человеком-моделью (гибрид товарка+fashion)

**Типы запросов к generate-image:**
1. **Fashion VTON** — основной (одежда + модель + поза + фон)
2. **Calibration** — калибровка модели (только модель без одежды)
3. **Photo Edit** — точечное редактирование существующего фото
4. **Product Mode** — предметная съёмка товаров
5. **Photoshoot** — пакетная генерация (3 или 5 кадров параллельно)

### Текущие источники данных, которые есть
1. **Vercel Logs** — серверные логи (console.log в generate-image.js): время запроса, тип операции, количество garment images, наличие refs, время генерации, ошибки
2. **Firebase Firestore** — пользователи, подписки, кредиты, платежи, сохранённые модели/локации
3. **Firebase Auth** — данные авторизации, количество пользователей
4. **KIE.ai API** — taskId, статус задачи (success/failed/timeout), время выполнения
5. **Google Cloud Monitoring** — расход токенов Gemini, стоимость по API-ключам

### Известные типы ошибок в системе
1. **KIE.ai timeout** — задача не завершилась за 5 минут (100 поллов × 3с)
2. **429 / RESOURCE_EXHAUSTED** — превышен rate-limit KIE.ai или Gemini
3. **422 / not supported** — модель генерации временно недоступна
4. **400 / INVALID_ARGUMENT** — некорректный запрос (слишком большое изображение и т.д.)
5. **Image download failure** — не удалось скачать результат с KIE.ai (сеть/таймаут)
6. **Firebase Storage upload error** — ошибка загрузки изображения в хранилище
7. **Payment webhook failures** — Telegram не доставил вебхук или Firestore не записал
8. **Subscription expiration edge cases** — пограничные ситуации при истечении подписки

## 🔴 В чём заключается проблема / вопрос

Сейчас у нас **нулевая видимость** в то, что реально происходит с продуктом в продакшене:
- Мы **не знаем**, сколько генераций делают пользователи, когда, какие типы, с какими настройками
- Мы **не видим** ошибки в реальном времени — узнаём о них только когда пользователь жалуется
- Мы **не можем** отследить конверсию: сколько пришло → попробовали → купили → используют регулярно
- Мы **не знаем** экономику: сколько реально стоит каждый пользователь, каков LTV, каков churn
- Мы **не контролируем** расход API: сколько денег тратится на KIE.ai и Gemini
- Мы **не можем** дать админу/оператору инструмент для мониторинга здоровья системы
- У нас нет централизованного лога генераций (с промптами, настройками, результатами, ошибками)
- У нас нет алертинга — если всё сломается в 3 ночи, мы узнаем утром из жалоб

## 🎯 Наша цель

Спроектировать **полноценную аналитическую админ-панель** для Seller Studio, которую можно будет давать администраторам и операторам для мониторинга всех процессов. Админка должна быть:

1. **Исчерпывающей** — показывать ВСЕ ключевые метрики, события и процессы, происходящие в системе
2. **Операционно полезной** — не просто графики ради графиков, а реальный инструмент для принятия решений
3. **Реалтаймовой** — данные обновляются в реальном времени или near-realtime
4. **Алертинговой** — оповещения при критических ситуациях (всплеск ошибок, падение API, подозрительная активность)
5. **Масштабируемой** — архитектура, которая не развалится при росте до 1000+ активных пользователей

### Примерный поток данных для 1 генерации
```
1. Клиент отправляет POST /api/generate-image с body:
   - garmentImageUrls[] или garmentImagesBase64[]
   - modelPreset, posePreset, cameraAngle, backgroundPreset
   - aspectRatio, attributes, isBeautyMode, biometricSeed
   - modelReferenceImages[], locationImages[]
   - isProductMode, categoryId, withHumanModel
   - isCalibration, isPhotoEdit, editInstruction

2. Сервер обрабатывает:
   - Скачивает garment images с Firebase Storage URLs
   - Определяет пол (detectGender)
   - Строит XML-структурированный промпт
   - Загружает base64 → KIE.ai File Upload API
   - Создаёт задачу KIE.ai (createTask)
   - Поллит результат (до 100 раз × 3 секунды)
   - Скачивает результат → base64 → отправляет клиенту

3. Возможные точки отказа:
   - Скачивание garment images
   - Загрузка в KIE.ai File Upload
   - Создание задачи KIE.ai
   - Поллинг (timeout 5 мин)
   - Скачивание результата
   - Ответ клиенту (Vercel maxDuration 120s)
```
